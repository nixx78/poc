# Rate Limiter — основные концепции

## Бизнес-цель
Что именно нужно ограничивать:  API/endpoint/сервисы ?

Зачем вводится ограничение?
* защита от DDoS
* fair usage
* тарифные планы (billing)

## Гранулярность лимитов

По какому ключу ограничиваем?
* userId
* IP
* API key

Нужны ли комбинированные лимиты, например: per user + per IP + per endpoint

## Характеристики лимита

Тип лимита:
* requests/sec (RPS)
* requests/min/hour/day
* Допускается ли burst (кратковременное превышение лимита) ?

## Поведение при превышении

Что делать при превышении лимита?
* HTTP 429 (Too Many Requests)
* отклонять запрос
* ставить в очередь

Нужно ли возвращать:
* Retry-After
* оставшийся лимит

## Тип Rate limit в зависимости от архитектуры системы

В случае распределенной системы, нужно решить как хранить состояние.
* Redis
* in-memory (не подходит для кластера)
* distributed cache (Hazelcast и т.д.)

## Требования к точности

Насколько строго нужно соблюдать лимит?
* строгий (0 превышений) - Sliding Window
* допускается погрешность - Fixed Window

## Производительность

Ожидаемый throughput (RPS), насколько важно влияние Rate Limiter на производительность системы
* Допустима ли дополнительная задержка?
* можно ли делать network call (например, в Redis)

## Отказоустойчивость

Что делать при падении хранилища (например, Redis)?
* fail-open (разрешать всё)
* fail-closed (блокировать всё)

## Консистентность

* Нужна ли строгая консистентность?
* Или допустима eventual consistency?

## Управление конфигурацией

Лимиты:
* статические (в коде)
* динамические (из БД / config server)
* Нужна ли поддержка динамического изменения конфигурации?

## Логирование и мониторинг

Нужно ли:
* делать запись в журнал при превышении лимита
* собирать метрики (Prometheus)
* настраивать уведомления

## Место внедрения Rate limiter в систему

Где реализуется Rate Limiter?
* API Gateway (предпочтительно)
* внутри сервиса (Filter / Interceptor)
* отдельный сервис
  

## Основные алгоритмы Rate Limiter
  
### Fixed Window (фиксированное окно)

Время делится на фиксированные интервалы (например, 1 минута)
Счётчик запросов обнуляется в начале каждого окна
Простая реализация (counter + TTL)

Плюсы:
* очень простой
* быстрый
* хорошо масштабируется

Минусы:
* “boundary problem” — всплеск на границе окна (например: 100 запросов в конце минуты + 100 в начале следующей)

### Sliding Window (скользящее окно)

Учитываются запросы за последний период времени, а не фиксированное окно
Обычно реализуется через:
список timestamp'ов или rolling counter

Плюсы:
* высокая точность
* нет скачков на границах

Минусы:
* сложнее реализация
* выше нагрузка (память/CPU)

### Token Bucket (ведро с токенами)

Есть “ведро” с токенами (ёмкость = max burst), при этом, токены добавляются с постоянной скорость.
Каждый запрос “забирает” токен, если токенов нет то запрос отклоняется

Плюсы:
* поддерживает burst (накопленные токены)
* гибкий контроль скорости
* широко используется

Минусы:
* немного сложнее, чем Fixed Window

### Leaky Bucket (протекающее ведро)

Запросы попадают в очередь (bucket) и обрабатываются с фиксированной скоростью.
Если очередь переполнена, то запросы отбрасываются

Плюсы:
* выравнивает нагрузку
* стабилизирует поток запросов

Минусы:
* не поддерживает burst (в классическом виде)
* возможны задержки (из-за очереди)
